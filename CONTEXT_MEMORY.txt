================================================================================
                        ARCHITECTURE / DECISIONS
================================================================================
(Key design decisions, patterns, tech stack choices, etc.)

- Multi-provider uses Strategy pattern + Registry. BaseProvider is the contract.
- Model strings use format "provider/model" (e.g., "openai/gpt-4o").
- Tool system uses BaseTool ABC + ToolRegistry. Tools self-describe via JSON Schema.
- Custom tools stored in DB, wrapped as DynamicTool instances, executed in sandboxed subprocess.
- Skills system uses OpenClaw SKILL.md format (YAML frontmatter + markdown instructions).
  Active skills are injected into agent system prompts automatically.
- ChatService has an agentic loop: LLM calls tools, sees results, loops up to 10x.
- Database: SQLite at backend/data/assitance.db (auto-created on startup).
- Frontend proxied to backend via Vite dev server proxy config.
- All CORS allowed in dev (allow_origins=["*"]).
- Settings stored in pydantic-settings (from .env), with runtime update via API.
- Persistent Agent system: Users can define specialized agents with specific system prompts and models.
- Knowledge Base uses ChromaDB for local vector storage. Documents are chunked and embedded on upload.
- Workflows use a DAG model: Workflow -> Nodes (trigger/action) -> Edges. WorkflowEngine traverses linearly for MVP.
- ReactFlow library used for visual node graph editor in the frontend.
- Agent status uses in-memory singleton + WebSocket broadcast pattern (no DB overhead).
  Frontend connects directly to backend port 8321 for agent status WS (bypasses Vite proxy).
- Redis used for real-time agent heartbeats, task queue state, delegation chain tracking.
  If Redis unavailable, system falls back to in-memory operation gracefully.
- Multi-Agent Orchestration: System Agent as master coordinator, @mention-selective routing,
  hybrid auto/manual delegation modes, delegation chains with depth limits and circular
  invocation detection, token budget per chain (100k default).
- Unified Channel Chat: P2P and LLM modes merged into single channel system. All communication
  goes through unified channels with orchestration mode (autonomous/manual) per channel.
- Three-Layer Heartbeat: Agent process heartbeat via Redis pub/sub, task execution heartbeat
  with progress tracking, delegation chain heartbeat (event-based). Recovery protocol with
  failure escalation after 3 consecutive failures.
- Human-in-the-Loop: Sensitive tools (code_executor, file_manager) are blocked by default until user approval.
- Context Pruning uses a small/cheap model (e.g., gemini-2.5-flash) for summarizing old context to save money.
- Audio implementation uses Gemini for STT and edge-tts for TTS to avoid proprietary OpenAI audio costs/keys.
- [2026-03-11] PostgreSQL + Alembic: Database upgraded from SQLite-only to dual support (PostgreSQL via asyncpg for production, aiosqlite for development). All manual ALTER TABLE migrations replaced with Alembic. Connection pooling with pool_size=20, max_overflow=10.
- [2026-03-11] Secret Protection: API keys encrypted at rest via Fernet (cryptography library). SecretManager singleton provides transparent encrypt/decrypt. Providers access keys only through SecretManager, never raw settings.
- [2026-03-11] Capability-Based Task Routing: OrchestrationEngine uses CapabilityRegistry.find_best_agent() with weighted scoring (40% domain expertise, 30% tool overlap, 30% success_rate) instead of name-based agent matching.
- [2026-03-11] Three-Layer Agent Memory: Working Memory (ephemeral DB, 1hr expiry), Episodic Memory (task summaries in DB), Long-Term Memory (ChromaDB vector embeddings). Brain service orchestrates all three layers.
- [2026-03-11] 6-Stage Task Lifecycle: PLAN → DECOMPOSE → ASSIGN → EXECUTE → VERIFY → FINALIZE. Each stage tracked in Task.lifecycle_stage for full observability.
- [2026-03-11] Distributed Task Queue: Redis ZADD sorted sets for priority (URGENT=0, NORMAL=5, LOW=10). SETNX distributed locking prevents duplicate consumption. Dead-letter queue for failed tasks.

### 1. Human-in-the-Loop (HITL) Approval Flows

To prevent your agent from running destructive Python code or sending unauthorized emails, you need an asynchronous pause mechanism in your backend that waits for a frontend signal.

**The Strategy:** Wrap sensitive tools with an approval flag. When triggered, the backend pauses execution using an `asyncio.Event`, emits a WebSocket payload to the frontend requesting permission, and waits.

**Backend Pseudo-code (FastAPI):**

```python
import asyncio

# A dictionary to hold pending approvals mapped by task ID
pending_approvals = {}

async def execute_tool_with_hitl(tool_name, arguments, task_id, ws_manager):
    if tool_requires_approval(tool_name):
        # Create an event lock
        approval_event = asyncio.Event()
        pending_approvals[task_id] = {"event": approval_event, "status": "pending"}
        
        # Emit WebSocket event to Frontend
        await ws_manager.send({"type": "APPROVAL_REQUIRED", "task_id": task_id, "tool": tool_name})
        
        # Pause execution until the frontend resolves the event
        await approval_event.wait()
        
        if pending_approvals[task_id]["status"] != "approved":
            return "Execution denied by user."
            
    # Proceed with actual tool execution
    return await run_sandbox_tool(tool_name, arguments)

# WebSocket Listener Endpoint
@app.websocket("/api-ws/agents/control")
async def websocket_control(websocket: WebSocket):
    data = await websocket.receive_json()
    if data["action"] in ["APPROVE", "DENY"]:
        task_id = data["task_id"]
        pending_approvals[task_id]["status"] = "approved" if data["action"] == "APPROVE" else "denied"
        pending_approvals[task_id]["event"].set() # Unblocks the execution

```

**Frontend Pseudo-code (Zustand + React):**

```typescript
// Zustand Store
const useAgentStore = create((set) => ({
  pendingApproval: null,
  setPendingApproval: (data) => set({ pendingApproval: data }),
  resolveApproval: (ws, taskId, action) => {
    ws.send(JSON.stringify({ action, task_id: taskId }));
    set({ pendingApproval: null });
  }
}));

// React Component
function ApprovalModal() {
  const { pendingApproval, resolveApproval } = useAgentStore();
  if (!pendingApproval) return null;

  return (
    <div className="glassmorphism-modal">
      <h3>Agent wants to run: {pendingApproval.tool}</h3>
      <button onClick={() => resolveApproval(ws, pendingApproval.task_id, "APPROVE")}>Allow</button>
      <button onClick={() => resolveApproval(ws, pendingApproval.task_id, "DENY")} className="bg-red-500">Deny</button>
    </div>
  );
}

```

### 2. Token Economy & Cost Tracking

When you have multiple agents delegating tasks, API rate limits are easily hit, and costs can skyrocket. You need a centralized middleware in your `ProviderRegistry` to catch the `usage` metadata from every LLM response.

**The Strategy:** Standardize the token tracking across Gemini, Claude, and OpenAI. Save this to your SQLite DB asynchronously and stream the running total to the frontend.

**Backend Pseudo-code (FastAPI / ProviderRegistry):**

```python
async def call_llm_provider(agent_config, messages):
    provider = ProviderRegistry.get(agent_config.provider)
    
    # Standardized response from any provider
    response, usage = await provider.generate(messages) 
    
    # Calculate cost based on provider pricing tiers
    cost = calculate_cost(agent_config.model, usage.prompt_tokens, usage.completion_tokens)
    
    # Async background task to update SQLite so it doesn't block the stream
    background_tasks.add_task(save_token_usage_to_db, agent_config.id, usage, cost)
    
    # Emit to frontend via WS for real-time dashboard updates
    await ws_manager.send({"type": "TOKEN_UPDATE", "agent_id": agent_config.id, "cost_added": cost})
    
    return response

```

### 3. Context Window Pruning & Memory Consolidation

A long-running Orchestrator agent will eventually fill its context window. You need a rolling memory pruner.

**The Strategy:** Before passing the `messages` array to the LLM, check the token length. If it exceeds a threshold (e.g., 80% of max context), extract the oldest $N$ messages (skipping the system prompt), send them to a cheaper/faster model to be summarized, and replace them with a single `system` or `assistant` summary block.

**Backend Pseudo-code (Python):**

```python
async def prune_context_if_needed(messages, max_tokens=8000):
    current_tokens = count_tokens(messages)
    
    if current_tokens < max_tokens:
        return messages
        
    # Keep the system prompt (index 0) and the most recent 5 messages
    system_prompt = messages[0]
    recent_messages = messages[-5:]
    messages_to_summarize = messages[1:-5]
    
    # Use a fast local model (Ollama) or cheaper cloud model for the summary
    summary = await generate_summary(messages_to_summarize)
    
    # Rebuild the array
    pruned_messages = [
        system_prompt,
        {"role": "system", "content": f"Summary of earlier conversation: {summary}"},
        *recent_messages
    ]
    
    return pruned_messages

```

### 4. Graceful Error Handling & "Self-Correction" Loops

Anyone who has spent time debugging multi-agent frameworks knows that tools fail constantly—syntax errors, network timeouts, or missing variables. The framework shouldn't crash; it should hand the error back to the LLM.

**The Strategy:** Wrap tool execution in a `try/except` block. Catch the traceback and return it as the tool's string output, appending instructions for the agent to fix the mistake.

**Backend Pseudo-code (Python):**

```python
import traceback

async def execute_tool_safely(tool_name, args, max_retries=3):
    for attempt in range(max_retries):
        try:
            # Attempt to run the python script or API call
            result = await run_tool(tool_name, args)
            return result 
        except Exception as e:
            error_trace = traceback.format_exc()
            
            # Formulate the error as feedback for the LLM
            correction_prompt = f"Tool '{tool_name}' failed with error:\n{error_trace}\nAnalyze the error and try a different approach."
            
            # Send this back to the LLM's message history to try again
            args = await ask_llm_for_correction(correction_prompt)
            
    return f"Task failed after {max_retries} attempts. Final error: {error_trace}"

```

### 5. Seamless Voice/Audio Input/Output

To make CrossClaw feel like a true assistant, you need to handle audio streams.

**The Strategy:** Use the browser's `MediaRecorder` API to capture audio, send the binary buffer over WebSockets to FastAPI, process it with a Speech-to-Text model (like Whisper), pass the text to the Orchestrator, and then pipe the Orchestrator's response text into a Text-to-Speech engine.

**Frontend Pseudo-code (React):**

```javascript
// Using MediaRecorder to capture audio
const startRecording = async () => {
  const stream = await navigator.mediaDevices.getUserMedia({ audio: true });
  const mediaRecorder = new MediaRecorder(stream);
  
  mediaRecorder.ondataavailable = (event) => {
    // Send audio blob directly to the FastAPI WebSocket
    if (event.data.size > 0) {
      ws.send(event.data); 
    }
  };
  mediaRecorder.start(1000); // chunk every second
};

```

**Backend Pseudo-code (FastAPI WS Endpoint):**

```python
@app.websocket("/ws/audio")
async def audio_endpoint(websocket: WebSocket):
    await websocket.accept()
    
    # Receive binary audio chunks from Vite frontend
    audio_bytes = await websocket.receive_bytes()
    
    # 1. Speech to Text (e.g., local Whisper or external API)
    user_text = await stt_service.transcribe(audio_bytes)
    
    # 2. Process via CrossClaw Orchestrator
    llm_response = await orchestrator.chat(user_text)
    
    # 3. Text to Speech (e.g., edge-tts or ElevenLabs)
    response_audio_bytes = await tts_service.synthesize(llm_response)
    
    # 4. Send audio back to frontend to play
    await websocket.send_bytes(response_audio_bytes)

```
